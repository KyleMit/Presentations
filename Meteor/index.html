Mongo v SQL - CRUD


SQL


SELECT _id, name, address -- Projection
FROM   users              -- Table
WHERE  age > 18           -- Criteria
LIMIT  5                  -- Cursor Modifier

Mongo


db.users.find(            // Collection
 { age: { $gt: 18 } },    // Criteria
 { name: 1, address: 1 }  // Projection
).limit(5)                // Cursor Modifier
  • Mongo CRUD
  • docs.mongodb.org/manual/core/read-operations-introduction/
  • db.collection.find(query [, projection])
  • Query and Projection Operators - Can't pass < symbol - so we pass a keyword operator instead
  • Order of operations makes so much more sense. First, where are we looking, then what subset are we using, then what are we grabbing, then how do we want that info returned

Mongo v SQL - CRUD


SQL


UPDATE users                 -- Table
SET    name = 'Ed', 		     -- Modify
       rel_status = 'Single' 
WHERE  id = 1                -- Criteria

Mongo


db.users.update(             // Collection
 { id: 1 },                  // Criteria
 { $set: {                   // Modify
 		name: 'Ed',
 		rel_status: 'Single'
  }}  
)
  • db.users.update(query,update,options);
  • Update Operators (begin with $)
    • $inc - increments the value of the field
    • $mul - multiplies the value of the field
    • $rename - renames the field (might want to call on every field, but there's nothing to enforce that documents in a collection have the same property names
    • $set - Sets the value of a field
    • $unset - Removes the value of a field
  • Operators begin with $

git co p3
  • body and head are both available as Template helpers
  • In this example we're creating a template named question-list and giving a property questions
  • Note: only things declared through this template helper are available to the template (with the exception of helper methods added to all templates).
    Which means, within the data context of the template helper we can't access the questions collection directly. We need to manually expose it. This is a good thing. Helps keep the scope of what the template small. Whenever a change is made to one of the template items, it can re-render the relevant DOM elements.

That's all you really need - much more than that and too much logic is going into the view. Do it with helpers. You might object: but that's fewer directives than angular provides: on the grounds that the more of something you can possibly have, the better it is. Yeah, but things like ng-src only existed to prevent the flash of unstyled content issues. There's a fundamental difference in ownership. In angular, HTML owns all the code. it's right there in the body. In meteor, meteor owns everything. So in angular, I put all that markup in regular HTML and then rely on angular kicking in at the last minute and saying, hey, there's a new element that matches my directive, I guess I'll do something about it. In meteor, you write your templates and meteor renders good old fashioned mark up.

We're going to do three things:
  1. grab the event info
  2. write to the database
  3. Clean up after ourselves

Template.question_submit.events({
	'submit form': function (event, template) {
		// grab the input
		var text = event.target.question.value;

		// write to the database
		Questions.insert({
			text: text
		});

		// clean up form events
		event.preventDefault();
		event.target.question.value = "";
	}
});

Insecure Package

Will someone open their console and run this:


Questions.remove({})

What Happened?

"Not permitted. Untrusted code may only remove documents by ID."

How about this instead?


Questions.find().forEach(function(doc) {
	Questions.remove(doc._id)
});
  • Okay, we've slowed down an attacker for about a quarter second
  • cusor.forEach
  • meteor list - list pacakages added to your project
  • View Packages in meteor-platform

in secure


meteor remove insecure

Two Solutions:

  1. Allow/Deny - Explicitly declare what is allowed
  2. Methods - Canonical method is run on server
  • You can use both interchangeably
  • Methods are always safe because you own the server code - but you still need to sanitze inputs Check(property, type)
  • Latency compensation - canonical database operation is always preformed on server. but the client will try to predict the outcome
  • Collection2

Allow / Deny

Allow


Posts.allow({
  insert: function (userId, doc) {
    // user must be logged in and modify their own doc
    return (userId && doc.owner === userId);
  }
});

Deny


Posts.deny({
  update: function (userId, docs, fields, modifier) {
    // can't change owners
    return _.contains(fields, 'owner');
  }
});
  • collection.allow
  • collection.deny
